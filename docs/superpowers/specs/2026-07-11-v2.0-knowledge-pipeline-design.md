# ADD v2.0 — 知识沉淀系统设计文档

> **版本**: v2.0-SPEC-1 | **日期**: 2026-07-11 | **前置版本**: v1.0 (P0-P3 Done, 94.3% recall, 96/96 TDD)

## 1. 设计意图

### 问题陈述

ADD v1.0 每次审计从零开始。v1.0 积累的数据——3 个 benchmark 项目（13 known bugs, TP/FP/FN 标注完整）、4 个 audit-log 案例、4 个 guards 模板、1 个 reference 规则文件、1 条 decisions 记录——全部处于 **数据孤岛** 状态。它们之间没有连接，不能互操作，不能自动转化为可复用知识。

### 设计目标

将 v1.0 的数据孤岛连接为 **知识生产流水线（Knowledge Pipeline）**，使 ADD 具备：

1. **自动提取**: 从 benchmark TP 标注和 reference 规则文件中提取可复用检测规则
2. **结构化映射**: 建立 spec 章节→代码模块→ADR 不变式的精确索引
3. **知识注入**: 在每次新审计的 subagent prompt 中注入历史积累的知识

### 差异化定位

| 工具 | 检测对象 | 知识形式 | ADD v2.0 差异化 |
|---|---|---|---|
| CodeQL | 代码缺陷 (pattern-based) | QL 查询库 | ADD 可积累 **spec-driven** 规则（代码是否忠于设计意图） |
| Semgrep | 代码模式 (AST pattern) | YAML 规则注册表 | ADD 规则覆盖设计对齐、契约验证、修正项追踪（纯代码工具不覆盖） |
| CodeGraph | 代码结构 (symbol relation) | SQLite 知识图谱 | ADD 图谱包含 **spec↔code 映射层**（纯代码工具不覆盖） |

## 3. 核查中发现的分析偏差 (7 项, 已修正)

本次 v2.0 分析经历了逐项代码核查，发现并修正了 7 个偏差：

1. **audit-log 案例数量高估**: 声称 5 个，实际 4 个 (其中 2 个为 pre-refactor 快照)。修正：rule_extractor 以 benchmark 数据为主力源 (12 TP)，非 audit-log。
2. **P2.7 (Dual-Subagent Voting) 被遗漏**: ROADMAP 唯一未 Done 的 P2 项。修正：纳入 v2.0 首期。
3. **spec_graph.py 工程量低估**: 声称 3 天，实际需 tree-sitter + ADR tracing。修正：v2.0 做映射层 (纯 Python stdlib)，v2.1 做 AST 集成。
4. **Benchmark→Guard 泛化能力高估**: 具体 bug instance → 通用规则是开放研究问题。修正：v2.0 只做结构化转换 (不尝试自动泛化)，泛化留给 v2.2 LLM 辅助。
5. **社区共享需要生态基础**: add-rules 独立仓库需要用户量。修正：v2.0 先做内部 formats 标准化，v2.3 才分离独立仓库 (需 ≥3 外部用户)。
6. **代码知识图谱与 CodeGraph 重叠**: CodeGraph 已做 tree-sitter → SQLite。修正：ADD 知识图谱聚焦 spec↔code 映射层，不重复代码结构索引。
7. **v2.0 范围过大**: 同时交付三种能力不切实际。修正：v2.0 聚焦知识沉淀管道的 **最小可行验证**。

## 4. 核心架构

```
  ┌──────────────────────────────────────────────────────────┐
  │              Knowledge Pipeline / 知识流水线               │
  │                                                            │
  │  Data Sources (v1.0)       Extractors (v2.0 new)           │
  │  ┌──────────────────┐    ┌────────────────────────┐       │
  │  │ benchmark/*.json ├───→│ rule_extractor.py      ├──┐    │
  │  │ (TP + evidence)   │    │ benchmark→guards.yml   │  │    │
  │  └──────────────────┘    └────────────────────────┘  │    │
  │                                                        │    │
  │  ┌──────────────────┐    ┌────────────────────────┐  │    │
  │  │ references/*.md  ├───→│ rule_extractor.py      │  │    │
  │  │ (semi-structured │    │ reference→guards.yml   │  │    │
  │  │  rules)           │    └────────────────────────┘  │    │
  │  └──────────────────┘                                  │    │
  │                                                        ▼    │
  │                                          ┌─────────────────┐│
  │                                          │ docs/audit/      ││
  │                                          │ guards.learned   ││
  │                                          │ .yml (v2.0)      ││
  │                                          └────────┬────────┘│
  │                                                   │         │
  │  ┌──────────────────┐    ┌────────────────────────┐│         │
  │  │ spec.md + ADRs   ├───→│ spec_graph.py         ││         │
  │  │ + source code     │    │ (upgraded rule_index) ││         │
  │  └──────────────────┘    └───────────┬────────────┘│         │
  │                                      │              │         │
  │                                      ▼              ▼         │
  │                           ┌──────────────────────────────┐   │
  │                           │ ADD Knowledge Index           │   │
  │                           │ → spec_graph.json             │   │
  │                           │ → guards.learned.yml          │   │
  │                           │ → scores.json (trend)         │   │
  │                           └──────────────┬───────────────┘   │
  │                                          │                    │
  │                                          ▼                    │
  │                           ┌──────────────────────────────┐   │
  │                           │ Subagent Prompt Augmentation   │   │
  │                           │ (RAG pattern: inject relevant   │   │
  │                           │  rules + graph into prompt)     │   │
  │                           └──────────────────────────────┘   │
  └──────────────────────────────────────────────────────────┘
```

**关键原则** (继承 ROADMAP §Design Constraints):

1. **所有提取器是确定性 Python 脚本** — 通过 RunCommand 执行，无 AI 参与
2. **AI 只在最后一步介入** — subagent prompt 中注入提取的知识，而非让 AI 生成知识
3. **规则人审 AI 用** — guards.learned.yml 标记 `source: auto-extracted`，人工 review 后移入 guards.project.yml
4. **失败不阻塞** — 所有提取器失败/超时 → 审计继续，不带增强知识

## 5. v2.0 交付物

### 5.1 rule_extractor.py — Benchmark → Guard 转换器

```
功能: 从 benchmark TP 标注和 reference 规则文件中提取可复用 guards
输入:
  --benchmark-dir <docs/benchmark/>
  --references-dir <references/>
  --template <guards.template.yml>
  --output <guards.learned.yml>
输出: 格式化的 guards.learned.yml, 每条 guard 标注来源

转换逻辑 (确定性, 无 AI):
  1. 遍历 benchmark-result.json 中的 tp_classification[]
  2. 对每个 TP:
     a. 提取 description + evidence.file + category + severity
     b. 从 evidence.file 推断 scope glob
     c. severity (P0/P1/P2) → guard severity (blocker/critical/warning)
     d. 生成 check 字段 + source 标注
  3. 遍历 references/ 中规则文件的 Rule N → Detection Pattern → Audit Checks
  4. 提取 check 指令, 生成 scope + severity, 标注 source
  5. 合并 + 去重 (description 相似度 >80%)
  6. 输出 guards.learned.yml

成本上限: 30s。最大产出: 50 条 guards。
```

**Benchmark TP → Guard 转换示例**:

```
Benchmark TP:
  known_bug_id: "KB-1"
  description: "HTTPDigest.__call__ raises exception even when auto_error=False"
  evidence: "fastapi/security/http.py:L415-L419"
  severity: "P0"
  category: "behavior"

输出 Guard:
  id: "LEARNED-FASTAPI-001"
  description: "HTTP authentication classes MUST respect auto_error parameter"
  scope: "**/security/**/*.py"
  severity: "blocker"
  check: "Verify all HTTP authentication classes check auto_error parameter
         before raising HTTPException. Compare with HTTPBearer pattern."
  source: "auto-extracted from benchmark/fastapi-0.115.8 KB-1 (TP, exact match)"
  confidence: "verified"
```

### 5.2 spec_graph.py — 升级版 rule_index.py

```
功能: 建立 spec 章节→代码模块→ADR 不变式的结构化映射
新增能力 (相比 rule_index.py):
  A. Module Mapping: "## Module: <name>" → 关联代码目录
  B. ADR Tracing: ADR "Decision:" 段落 → grep 确认代码证据
  C. Corrective Linking: spec "修正 X.Y" → 代码中相关行
  D. Guard Binding: guards.learned.yml scope → 文件列表

输入:
  --spec <spec.md>           --adrs <adr-dir>
  --guards <guards.yml>       --repo <path>
  --output <spec_graph.json>

输出 spec_graph.json 结构:
{
  "project": "<name>", "version": "<version>",
  "spec_sections": [
    {
      "section": "## Phase 1: Spec Inventory",
      "module": "SKILL.md",
      "constraints": ["..."],
      "code_locations": ["SKILL.md:L181"]
    }
  ],
  "adr_decisions": [
    {
      "adr": "docs/ROADMAP.md",
      "decision": "P0 items must be fixed before P1",
      "code_evidence": [
        {"file": "issues_tracker.py", "line": 4,
         "match": "open → in_progress → fixed → verified"}
      ]
    }
  ],
  "guard_mappings": [
    {
      "guard_id": "LEARNED-FASTAPI-001",
      "scope_glob": "**/security/**/*.py",
      "resolved_files": ["fastapi/security/http.py"],
      "audit_lens": "behavior"
    }
  ]
}

保留 rule_index.py 关键词索引作为子功能。
成本上限: 30s。
```

**升级点对比**:

| 维度 | rule_index.py (v1.0) | spec_graph.py (v2.0) |
|---|---|---|
| 约束提取 | regex: must/shall/threshold | 保留 + ADR Decision 段落提取 |
| 模块映射 | 无 | spec 章节→模块→文件 glob |
| ADR 绑定 | keyword 索引 | Decision → grep evidence → line |
| Guard 集成 | 无 | guard scope → 文件 glob 解析 |
| 输出格式 | keyword→constraint list | 结构化 JSON 图谱 |

### 5.3 SKILL.md 新增章节 — Knowledge Loading

在 SKILL.md 中 Phase 0 与 Phase 1 之间插入:

```
## Knowledge Loading / 知识加载 (v2.0+)

### Step 0.5: Load Accumulated Knowledge

Before dispatching any subagent, load structured knowledge from
previous audits and reference files:

1. spec_graph.json (if available):
   → Inject into subagent prompt: spec-section→file→line mappings.
   → Reduces subagent context-switching cost.

2. guards.learned.yml (if available):
   → Merge with project guards.yml via merge_guards.py.
   → Learned guards marked source: auto-extracted.
   → Subagent checks them alongside human-authored guards.

3. scores.json (if available):
   → Include trend chart in audit report footer.
   → Highlight declining modules for deep audit.

If any knowledge file is missing: skip, proceed with normal audit.
If loading fails (>30s): skip, warn, proceed.
Knowledge is an accelerator, not a prerequisite.
```

### 5.4 P2.7 — Inter-Rater Reliability (v1.0 遗留)

```
功能: 使用 spec_graph 的 spec↔code 映射评估两个 subagent 的一致性
实现:
  1. Subagent A = Design Aligner (contract lens)
  2. Subagent B = Corrective Tracker (corrective lens)
  3. 从 spec_graph.json 提取共同覆盖的 spec sections
  4. 在共同覆盖区域计算 agreement rate + Cohen's kappa
  5. agreement < 0.5 → 标记 "possible knowledge gap"

与无知识图谱的区别:
  无 spec_graph: 两个 subagent 可能审查不同文件 → 不可比较
  有 spec_graph: 两个 subagent 在共同映射上审查 → 可比较

成本: +1 subagent dispatch。仅当 --inter-rater 启用 (非默认)。
```

## 9. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| rule_extractor 产出的 guard scope 过宽/过窄 | 中 | 中 — 降低 precision | learned guard 标记 confidence:auto-extracted, subagent 降低其权重; 人工 review 后升级 |
