### diff_miner.py — 从开源代码中学习 Bug 模式（Detection Track）

```
功能: 从 git history 中挖掘常见 bug-fix 变更模式
输入:
  --repo <path>                (被审计项目的 git 仓库)
  --since <date>               (挖掘起始日期)
  --output <pattern_candidates.json>
逻辑:
  1. git log --grep="fix|bug|CVE|vuln|patch|security" --oneline --since=<date>
  2. 对每个 commit: git show --stat → 获取变更文件列表
  3. 对高频变更模式做统计聚类:
     - 同一文件被 "fix" 提交修改 3+ 次 → flag 为 "hotspot"
     - 同一类修复模式出现 5+ 次 (如 "add null check") → 建议生成 guard
     - 同一类 CVE 类型出现 2+ 次 → 建议生成 security guard
  4. 输出 pattern_candidates.json:
     {"hotspots": [...], "patterns": [...], "suggested_guards": [...]}

成本上限: 60s。只分析最近 200 个 fix commit (避免全量扫描)。
```

### fix_history.py — 奖励信号驱动的修复匹配（Repair Track）

```
功能: 新发现匹配历史已验证修复 → 自动注入 fix_suggestion
输入:
  --findings <new_issues.json>
  --history-dir <docs/audit/>
  --output <augmented_issues.json>
逻辑:
  1. 加载 scores.json → 读趋势:
     ▲ (改善) → difflib 阈值 0.60 (宽松多复用)
     ▼ (退化) → difflib 阈值 0.80 (严格避免无效方案)
     ─ (稳定) → 阈值 0.65 (默认)
  2. 加载所有历史 issues-*.json → 只保留 status=verified
  3. 对每个新 finding:
     a. 构造搜索文本: category + severity + claim
     b. difflib.SequenceMatcher 与所有历史 claim 计算相似度
     c. 若 > 阈值 → 注入历史 fix_suggestion 为 suggested_fix
     d. 标注 match_score + source_history_id
  4. 输出 augmented issues JSON

成本上限: 30s。最大历史: 50 条 verified finding。
```

### 奖励信号闭环

```
fix_history.py 读取 scores.json 趋势
        │
        ├── ▲ (改善) → fixed_threshold = 0.60
        │              → "当前策略有效，多复用"
        │
        ├── ▼ (退化) → fixed_threshold = 0.80
        │              → "当前策略退化，严格匹配"
        │
        └── ─ (稳定) → fixed_threshold = 0.65
                       → 默认行为

连续 3 次 ▲ → 触发 "confident mode": 阈值额外降低 0.05
连续 3 次 ▼ → 触发 "deep-audit": 自动增加 2 个 extension lens
```

### v2.1 成功标准

| 指标 | 目标 | 测量方式 |
|---|---|---|
| diff_miner 热点检测 | 被审计项目 ≥2 hotspots 识别 | 检测到 ≥2 个曾被多次修复的文件 |
| fix_history 匹配率 | ≥40%（≥5/12 finding 有推荐） | 运行 fix_history on v2.0 self-audit findings |

## 3. v2.2 — 双轨合并：LLM 辅助泛化 + Fix Recipe Book

**目标**: 将 v2.1 积累的"具体模式"（specific bug instance）泛化为"通用规则"（general detection/fix rule）。两个独立的 Detection/Repair track 在 LLM 泛化层合并，共享同一个泛化引擎。

### v2.2 系统架构

```
v2.1 产出:
  diff_miner → pattern_candidates (具体变更模式)
  rule_suggester → rule_suggestions (建议规则)
  fix_history → verified_fixes (已验证修复方案)
        │
        ▼
┌──────────────────────────────────────────────────────────────────┐
│                  LLM Generalization Engine                        │
│                  (LLM 辅助：具体→通用规则泛化)                       │
│                                                                   │
│  Prompt: "以下是 N 个 bug-fix 具体实例。抽象出通用的检测规则和     │
│           修复配方。"                                              │
│                                                                   │
│  Input layers (由具体到抽象):                                      │
│    L1: 具体 fix commit diff → 提取变更模式                         │
│    L2: 具体 bug description → 抽象为 guard check 指令               │
│    L3: 具体 fix_suggestion → 抽象为 fix recipe                     │
│                                                                   │
│  Output: rules.generalized.yml + recipes.generalized.yml           │
└──────────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────┐
│  Rule Scoring & Validation            │
│  每条泛化规则在 benchmark 上复测 recall │
│  recall ≥ 80% → verified              │
│  recall < 80% → flagged for review     │
└───────────────────────────────────────┘
```

| 文件 | 行数 | 功能 |
|---|---|---|

| `docs/audit/rules.generalized.yml` | 新数据 | LLM 泛化后的通用检测规则 |
| `docs/audit/recipes.generalized.yml` | 新数据 | LLM 泛化后的修复配方库 |
| `scripts/rule_validator.py` | ~80 | 泛化规则在 benchmark 上复测 recall |
| `scripts/fix_recipes.py` | ~120 | 机制 A: 人类维护的配方书 + LLM 导入 |
| `SKILL.md` (+Step 3.6) | +15 | Generalized Rule Injection |

### fix_recipes.py — 机制 A 实现

```
功能: 人类维护 + LLM 导入的双源修复配方库
配方格式 (recipes.generalized.yml):
  - id: "FIX-NULL-DICT-GET"
    pattern: "dict.get('key', default) returns None when JSON value is null"
    fix:
      steps: ["Change .get('key', 'default') to .get('key') or 'default'"]
      regression_risk: "none"
      verification: "grep -rn '\.get(' affected_files"
    confidence: "verified"  # LLM泛化 + benchmark验证
    source: "v2.0 self-audit ADD-P0-1 + LLM generalization"

固定内置配方 (human-authored):
  NULL-DICT-GET, TRACEBACK-LOSS, EVIDENCE-PARSING, SCOPE-GLOB, RETURNCODE-CHECK

LLM 贡献配方 (v2.2+):
  从 diff_miner 的模式 + fix_history 的修复方案 → LLM 泛化 → 自动候选
```

### mcp_lookup.py — 机制 B 完整实现

```
功能: 外部知识注入管道——MCP 搜索→缓存→引用持久化
输入:
  --findings <issues.json>       (新审计发现，优先处理 P0+P1 无 fix 建议的)
  --references-dir <references/>
  --cache-dir <.mcp_cache/>
逻辑:
  1. 遍历 issues.json → 筛选 severity=P0/P1 且无 suggested_fix 的 finding
  2. 对每个 finding:
     a. 构造查询: "{category} {claim}" + 被审计项目语言
     b. 顺序尝试: english-search → stackexchange → paper-search
     c. 缓存搜索结果 (.mcp_cache/ → 24h TTL)
     d. 若找到匹配 → 提取为结构化 reference → 写入 references/
     e. 若 3 个 source 均无结果 → skip (静默)
  3. 输出 mcp_results.json + 更新的 references/ 文件
  4. 在 audit report 底部标注: "MCP lookup: N findings augmented"

成本上限: 60s。最多 5 个 finding 做 MCP lookup (避免 overfetch)。
MCP 调用必须通过 run_mcp tool。
```

## 6. 完整知识闭环：从审计到自我进化

```
                        ┌───────────────────────────────────────┐
                        │         ADD Knowledge Pipeline         │
                        │                                       │
    New Audit ──────►   │  Detection Phase (Phases 1-3)         │
                        │    ├── guard injection (learned+generalized)│
                        │    ├── spec_graph knowledge preload    │
                        │    └── 7 lens parallel subagents       │
                        │              │                        │
                        │              ▼                        │
                        │  Repair Phase (Phases 4-6)             │
                        │    ├── fix_history: match→suggest     │
                        │    ├── fix_recipes: pattern→apply     │
                        │    └── mcp_lookup: P0 gap→research   │
                        │              │                        │
                        │              ▼                        │
                        │  Knowledge Update (post-audit)         │
                        │    ├── rule_extractor: learn→guards   │
                        │    ├── fix_history: verified→store    │
                        │    ├── spec_graph: rebuild index      │
                        │    └── scores.json: record trend      │
                        │              │                        │
                        │              ▼                        │
  Next Audit ◄───────── │  Feedback Loop                        │
                        │    ├── reward signal: adjust threshold │
                        │    ├── rule_suggester: suggest new    │
                        │    ├── LLM generalization: pattern→rule│
                        │    └── benchmark_syncer: pull new data │
                        └───────────────────────────────────────┘
```

**核心进化逻辑**：

1. **审计** (每次) → 发现新问题
2. **修复** (每次) → 积累 fix_suggestion 知识
3. **更新** (每次) → guards.learned.yml + fix_history + spec_graph 跟着进化
4. **反馈** (每次) → scores.json 趋势调整阈值 → 越匹配越自信，越偏离越严格
5. **泛化** (每版本) → LLM 辅助将具体经验升华为通用知识
6. **对标** (每季度) → 从开源标杆项目中获取新知识注入
