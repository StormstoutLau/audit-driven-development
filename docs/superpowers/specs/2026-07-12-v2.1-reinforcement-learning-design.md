# ADD v2.1 — 强化-自主学习修复能力设计文档

> **版本**: v2.1-SPEC-1 | **日期**: 2026-07-12 | **前置版本**: v2.0.1 (97/100 A+)

## 1. 问题陈述

v2.0 实现了完整的审计→修复→再审计闭环：6 透镜审计发现了 12 个问题 (4 P0 + 4 P1 + 4 P2)，全部修复后评分从 F(0) 升至 A+(97)。但整个修复过程是**一次性人工知识**——没有一个机制能让 ADD 从这次修复中学到东西并应用到下一次审计中。

**具体痛点**：
- 同一个 bug 模式（如 `parse_evidence` null 崩溃）会在以后的新脚本中再次出现，因为未沉淀为可复用规则
- 修复方案的质量依赖人类判断，没有历史参考
- 6 个透镜的审计发现 + 修复方案之间互不知晓——Contract Guardian 不知道 Error Handler 找到了什么

## 3. 三种强化-自主学习机制分析

### 机制 A：Fix Pattern Recipe Book（修复模式配方书）

**核心思路**：将"发现 X → 修复 Y"的过程沉淀为可复用的 fix-recipe。

**数据来源**：
- `docs/audit/issues-*.json` 的 `fix_suggestion` + `regression_risk` + `verification_command`
- `git diff` 中实际修复的代码变化

**可行性**：✅ 中高。结构化的 fix_suggestion schema (P2.13) 已经定义：
```json
{
  "steps": ["Step 1: ...", "Step 2: ..."],
  "affected_files": ["file.py:L123-L145"],
  "regression_risk": "low",
  "verification_command": "pytest tests/...",
  "impact": "...",
  "mitigation": "..."
}
```

**代码量**：~80 行（新增 `scripts/fix_recipes.py`）

**实际例子**（来自 v2.0 自审）：
- Recipe `NULL-GUARD-PATTERN`: 任何 `dict.get("key", default)` 在 JSON key 为 `null` 时返回 `None` → 改 `or default`
- Recipe `EVIDENCE-EXTRACTION-PATTERN`: parse 文本中嵌入的 file:line → 用 `re.search(r'(?:at|in|evidence|file)\s+([\w./-]+\.py)...')`
- Recipe `TRACEBACK-PATTERN`: 顶层 except 丢失 traceback → 添加 `import traceback; traceback.print_exc()`

### 机制 B：MCP External Knowledge Feedback（外部知识反馈）

**核心思路**：审计发现问题后，主动用 MCP 工具（mcp_paper-search, mcp_english-search）搜索专家知识，带回最佳修复方案。

**数据来源**：
- MCP 搜索工具（学术论文、GitHub PR、StackExchange）
- `references/` 目录的知识文件

**可行性**：✅ 中低。MCP lookup 已经在 KB-5 (AfterValidator) 上验证过——那次使用 MCP 搜索找到了 FastAPI PR #13442，最终转化为 `references/python-pydantic-audit-rules.md`。但搜索质量取决于 query 构建，不是每次都能找到精确答案。

**代码量**：~40 行（在审计流程中新增 MCP lookup step）

**实际例子**：
- 发现 P0-4（evidence format inconsistency）→ MCP 搜索 "benchmark data format standardization json api"
- 发现 P1-4（code duplication）→ MCP 搜索 "python shared utility module best practices" → StackExchange 结果

### 机制 C：History-Guided Repair（历史引导修复）

**核心思路**：新发现自动匹配历史发现，复用已验证的修复方案。

**数据来源**：
- `docs/audit/issues-*.json`（所有已修复的 finding）
- `docs/audit/scores.json`（评分趋势）
- `git log`（实际修复提交）

**可行性**：✅ 高。实现原理类似于 CodeQL 的"已知模式查询"——将新发现与历史发现的 `claim`、`category`、`severity` 进行语义匹配，若相似度 > 0.7，则注入对应的 fix_suggestion 作为参考。

**代码量**：~120 行（`scripts/fix_history.py`）

**实际例子**：
- 新发现: "X.py 的 top-level except 丢弃 traceback" → 匹配历史 finding ADD-P1-2
  → 注入: `fix_suggestion: {steps: ["import traceback", "traceback.print_exc()"], affected_files: ["X.py:L303"], regression_risk: "none", verification_command: "python scripts/test_v20_consistency.py"}`

## 4. 推荐方案：Fix History Loop（最小代码变更，最大收益）

**选择理由**：机制 C（History-Guided Repair）需要的代码量最小（~120 行），但与现有基础设施（issues JSON + fix_suggestion schema + guards learned）的契合度最高。可以直接作为 `guards.learned.yml` 的补充——learned guards 是"检测规则"，fix recipes 是"修复知识"。两者结合形成完整的 Detections→Repairs 知识图谱。

不推荐单独实现机制 A/B 的原因：
- 机制 A（Recipe Book）需要大量手动维护配方 → 适合 v2.2 当积累 30+ recipes 时
- 机制 B（MCP Lookup）在外网访问不稳定时失败率高 → 适合作为可选加速器

### 最小实现：`scripts/fix_history.py`（~120 行）

```
功能：将新发现的 finding 与历史已修复 finding 进行语义匹配
输入：
  --findings <new_issues.json>           (新审计发现)
  --history-dir <docs/audit/>            (历史 issues-*.json)
  --output <augmented_issues.json>       (带历史修复建议的 issues)
逻辑：
  1. 加载所有历史 issues-*.json
  2. 只保留 status=verified 的历史 finding
  3. 对每个新 finding：
     a. 提取 claim 的语义关键词 (category + severity + claim tokens)
     b. 与历史 finding 做 difflib 相似度计算
     c. 若相似度 > 0.65 → 注入历史 fix_suggestion 作为 "suggested_fix" 字段
     d. 标注相似度 + 来源历史 ID
  4. 输出 augmented issues JSON (原有 finding + suggested_fix)

成本上限：30s。最大匹配：50 条历史 finding。
```

**与现有管道的集成点**：

```
Step 3 (Aggregate findings)
    ↓
fix_history.py --findings new_issues.json → augmented_issues.json
    ↓
Step 4 (Fix Baseline) — issues.json 现在每个 finding 可能携带 suggested_fix
    ↓
Step 5 (Fix P0) — 人工修复时可以查看 AI 推荐的历史修复方案
```

### 评分闭环：Reward Signal

利用现有的 `scores.json` 趋势作为强化学习的**奖励信号**：

```
如果 scores.json 趋势 ▲ → 当前修复策略有效 → fix_history 增加权重
如果 scores.json 趋势 ▼ → 当前修复策略退化 → fix_history 降低权重 → 触发 deep-audit
```

实现方式：fix_history.py 读取 scores.json，根据趋势（▲/▼/─）自动调整相似度阈值：
- ▲ 趋势（改善中）→ 阈值 0.60（宽松，多复用历史方案）
- ▼ 趋势（退化中）→ 阈值 0.80（严格，避免复用无效方案）
- ─ 趋势（稳定）→ 阈值 0.65（默认）

### 外部知识增强（轻量 MCP Hook）

在 Step 3.5 之后，可选调用 MCP 搜索作为 external knowledge source：

```
Step 3.5: Fix History Augmentation (fix_history.py)
    ↓
Step 3.6: (OPT-IN) External Knowledge Lookup
    if --mcp-lookup enabled:
        for each P0 finding without suggested_fix:
            construct MCP query from claim + category + file path
            search mcp_english-search / mcp_paper-search
            if result found → append to suggested_fix
```

这是机制 B 的轻量实现——不作为默认行为，需要 `--mcp-lookup` 显式开启。

### 与 CodeQL/Semgrep 的差异化

| 工具 | 检测知识 | 修复知识 |
|---|---|---|
| CodeQL | QL 查询库（检测规则） | 无内置修复建议 |
| Semgrep | YAML 规则注册表 | Semgrep Assistant（PR diff→规则） |
| **ADD v2.1** | guards.learned.yml（检测） + **fix_history.py（修复）** | 历史 finding→新 finding 语义匹配 + MCP 外部知识 |
