## 2. 模块审查结果

### 2.1 rule_extractor.py (评分: C+)

#### 阻断性问题 (P0 × 2)
- **P0-1**: parse_evidence() 在 evidence 值为 null 时 re.match(pattern, None) 崩溃 (EH-001)
  - 位置: rule_extractor.py L62 — `match.get("evidence", "")` 在 JSON evidence: null 时返回 None 而非 ""
  - 影响: 损坏的 benchmark JSON 导致整个 audit 管道崩溃，而非跳过单个项目
  - 修复成本: 1 行 — `evidence_text = match.get("evidence") or ""`

- **P0-4**: 9/12 benchmark TP 的 guards scope 为 `**/unknown.py` (CG-001, CT-06)
  - 位置: guards.learned.yml L12, L22, L32, L42, L52, L62, L72, L82, L92
  - 根因: Flask/FastAPI benchmark 的 evidence 不是 `file:line — desc` 格式，嵌入在自由文本中
  - 影响: 75% 的 learned guards 对下游审计无实际约束力
  - 修复成本: refactor — parse_evidence 需要能从嵌入文本中提取文件路径

#### 严重问题 (P1 × 1)
- **P1-2**: 顶层 except 丢弃 traceback (EH-002)
  - 位置: rule_extractor.py L303-L305
  - 影响: 生产环境调试不可能

#### 中等问题 (P2 × 1)
- **P2-1**: category 字段在 YAML 输出时丢失 (CG-002)
  - 位置: rule_extractor.py L295-L305 — main() 写 id/description/scope/severity/check 但不写 category
  - 影响: spec_graph guard binding 始终获得默认 category='behavior'
