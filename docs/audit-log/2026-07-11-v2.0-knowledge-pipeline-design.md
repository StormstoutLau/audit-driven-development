## 2. Background / 背景

ADD v1.0 reached 94.3% benchmark recall, 96/96 TDD PASS, 7 lenses, 15 scripts. However, all accumulated data — 3 benchmark projects (13 known bugs, 12 TP), 4 audit-log cases, 4 guards, 1 reference rule file, 1 decision record — existed as data silos. Each audit started from zero.

## 3. Design Decisions / 设计决策

### DD-1: Extractor→Injector architecture over monolithic knowledge engine

- **Decision**: Separate knowledge extraction (deterministic Python scripts) from knowledge injection (subagent prompt augmentation).
- **Rationale**: Extraction must be deterministic and auditable; injection can be AI-assisted. Separating them enforces the "human reviews rules, AI checks compliance" principle.
- **Alternatives considered**: Single monolithic knowledge engine → rejected because it would blur the deterministic/AI boundary.

### DD-2: Benchmark data as primary knowledge source (not audit-log)

- **Decision**: rule_extractor.py uses benchmark TP annotations (12 verified TP) as primary source, not audit-log free-text.
- **Rationale**: Benchmark data has structured TP/FP/FN, evidence pointers, and verified recall. Audit-log is Markdown free-text with no machine-readable findings.
- **Code review finding**: Previous analysis claimed "5 audit-log cases". Actual: 4 cases, 2 are pre-refactor snapshots.

### DD-3: spec_graph.py extends rule_index.py (not replaces it)

- **Decision**: spec_graph.py reuses rule_index.py's extract_constraints() and index_rules() functions, adding Module Mapping, ADR Tracing, and Guard Binding.
- **Rationale**: Backward compatibility — existing audits depending on rule_index.py keyword output continue working. New audits get the full graph.

### DD-4: Cohen's kappa at per-file granularity for P2.7

- **Decision**: Inter-rater reliability computes kappa on per-file finding presence (did both subagents find issues in the same files), not per-line finding overlap.
- **Rationale**: Per-file is the practical first step. Per-line requires exact evidence pointer alignment which depends on subagent prompt precision — this is precisely what P2.7 aims to measure.

### DD-5: Learned guards stay auto-extracted until human review

- **Decision**: guards.learned.yml entries carry `confidence: auto-extracted`. They are NOT auto-applied with full penalty weight. Subagent reduces their weight. Human must move them to guards.project.yml to upgrade to `verified`.
- **Rationale**: Preserves "human authors rules, AI checks compliance" principle from ROADMAP §Design Constraints.

### DD-6: Knowledge files are optional accelerators (not prerequisites)

- **Decision**: All knowledge files (spec_graph.json, guards.learned.yml, scores.json) are `skip on missing, skip on timeout` with no audit blocking.
- **Rationale**: ROADMAP §Design Constraint 4: "Pluggable, not replacement." v1.0 functionality works identically without any knowledge files.

## 4. Architecture Evolution / 架构演变

### v1.0 → v2.0 Architecture Changes

```
v1.0:                                      v2.0:
                                           
  Spec + ADRs ──→ Phase 0 (Gate)             Spec + ADRs ──→ spec_graph.py ──→ spec_graph.json
                     │                                          │
                     ▼                                          ▼
                   Phase 1 (Inventory)      Benchmark TP ──→ rule_extractor.py ──→ guards.learned.yml
                     │                                          │
                     ▼                                          ▼
                   Phase 2 (Audit)           ┌──── Step 0.5: Knowledge Loading ────┐
                     │                      │  spec_graph.json + guards.learned   │
                     ▼                      │  + scores.json → inject into prompt  │
                   Phase 3-6                 └──────────────────────────────────────┘
                                                     │
                                                     ▼
                                               Phase 1-6 (normal audit,
                                               with augmented subagent prompts)
```

## 5. Review Findings / 审查发现 (偏差修正)

本次 v2.0 设计经历了逐项代码核查，发现并修正了 7 个分析偏差（详见 spec §3）：
1. audit-log 案例数量 5→4
2. P2.7 被遗漏 → 纳入 v2.0
3. spec_graph 工程量 3天→拆分为两阶段
4. Benchmark→Guard 泛化能力高估 → v2.0 只做结构化转换
5. 社区共享无生态基础 → 推迟到 v2.3
6. 知识图谱与 CodeGraph 重叠 → 聚焦 spec↔code 映射
7. v2.0 范围过大 → 聚焦最小可行验证
